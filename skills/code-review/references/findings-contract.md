# Code-review findings contract

Every reviewer persona in this skill, and the `code-review-lead` that merges their work, obey this
contract. The lead passes its absolute path to each persona at dispatch; read it before reviewing.

You are a **read-only** reviewer. Inspect the diff and surrounding code and **return** your findings —
do not edit project files, run fixes, commit, push, or post anything.

## Severity scale (P0–P3)

| Level | Meaning | Action |
|-------|---------|--------|
| **P0** | Critical breakage, exploitable vulnerability, data loss/corruption | Must fix before merge |
| **P1** | High-impact defect likely hit in normal usage, breaking a contract | Should fix |
| **P2** | Moderate issue with meaningful downside (edge case, perf regression, maintainability trap) | Fix if straightforward |
| **P3** | Low-impact, narrow scope, minor improvement | Author's discretion |

Severity orders urgency. It is independent of confidence: a P2 can be anchor 100 (airtight but minor);
a P0 can be anchor 50 (critical but unconfirmed).

## Confidence anchors (0 / 25 / 50 / 75 / 100)

Pick the single anchor whose behavioral criterion you can honestly self-apply. Only these five integers
are valid — never a value between them, never a float, never `"high"`.

- **0 — Not confident.** A false positive that fails light scrutiny, or a pre-existing issue this change
  did not introduce. **Do not emit — suppress.**
- **25 — Somewhat confident.** Might be real, might be a false positive; you could not verify from the
  diff and surrounding code. **Do not emit — suppress.** Gather more evidence to reach 50+, or drop it.
- **50 — Moderately confident.** You verified it is real but it is a nitpick, narrow edge case, or has
  minimal practical impact. Style/subjective improvements land here. Surfaces only when the lead routes
  it to a soft bucket, or when it is P0 (critical-but-uncertain is never silently dropped).
- **75 — Highly confident.** You double-checked the diff and surrounding code and confirmed the issue
  affects users, downstream callers, or runtime behavior in normal usage. Requires naming a **concrete
  observable consequence** — a wrong result, an unhandled error path, a contract mismatch, a security
  exposure, a real missing-test scenario. "This could be cleaner" does not meet this bar; that is 50.
- **100 — Absolutely certain.** Verifiable from the code itself — compile/type error, definitive logic
  bug (off-by-one, swapped args, wrong return type), or an explicit standards violation with a quotable
  rule. No interpretation required.

The actionable floor is **75**. Emit nothing you cannot honestly anchor at 50+, and 50 only survives as
a P0 or a soft-bucket item. If your persona sets a stricter floor, honor it.

### Quote-the-line gate (kills the "symbol doesn't exist" false-positive class)

Before anchoring at **75 or 100**, quote the verbatim line(s) that make the finding true, with
`file:line`, as the **first item of `evidence`**. "field X doesn't exist" → quote the class/Meta/
migration where X would be defined. "may return None" → quote the initialization. "race between A and B"
→ quote both. **If you cannot quote the motivating line, you cannot claim 75+ — step down to 50.** When
a symbol is generated by a framework metaclass, ORM `Meta`, decorator, or migration, quoting the
meta-construct that creates it satisfies the gate; a failed `grep` for the literal name does not.

## Diff-scope tiers

Classify every finding by its relationship to the diff:

- **Primary** — lines added/modified in the diff. Your main focus; report at full confidence.
- **Secondary** — unchanged code in the same function/block that interacts with a changed line. Report,
  noting the issue lives in the interaction between new and existing code.
- **Pre-existing** — issues in unchanged code the diff neither touched nor made newly relevant. Set
  `pre_existing: true`. Reported separately; does not count toward the verdict.

Rule of thumb: if you'd flag the same issue on an identical diff that didn't include the surrounding
file, it's pre-existing. If the diff makes it *newly relevant* (a new caller hits an existing bug),
it's secondary.

### Scope mode

The lead passes a `scope_mode`:

- **`local`** — the working tree *is* the reviewed code. Read/Grep/`git blame` workspace files freely.
- **`remote`** — the working tree is **not** the reviewed head. Do **not** Read/Grep workspace paths for
  changed files; inspect via `git show <head-ref>:<path>` (the lead provides `<head-ref>`) or the diff
  hunks only. Local workspace contents are not evidence for findings on changed files.

## False positives — suppress entirely (not even at 25/50)

- **Pre-existing issues unrelated to this diff** (mark `pre_existing` only for unchanged, untouched code).
- **Pedantic style a linter/formatter already catches** — semicolons, indentation, import order.
- **Code that looks wrong but is intentional** — check comments, commit messages, PR body, surrounding
  code for intent first.
- **Issues already handled elsewhere** — a guard, middleware, framework default, or parallel handler
  upstream. Check callers and guards before flagging.
- **Restatements of what the code already does**, or "consider adding X" where X already exists.
- **Generic "consider adding" advice with no named failure mode** — if you can't say what breaks, drop it.
- **Code with a relevant lint-disable comment** for the exact rule you'd flag.
- **General code-quality opinions not codified in a standards file** — "this file is long," "too many
  params" — unless a project standard sets the rule.
- **Speculative future-work** — "might break under load," "what if requirements change" — without
  concrete current evidence the concern is reachable now.

## Advisory routing

If the honest answer to "what breaks if we don't fix this?" is "nothing breaks, but…", the finding is
advisory: set `autofix_class: advisory`, `confidence: 50`. The lead routes it to a soft bucket rather
than surfacing it as a primary action item. Do not suppress it — it has value; it just doesn't warrant a
decision. (The false-positive catalog above is stricter and wins: an FP-catalog shape is suppressed
entirely, never routed to advisory.)

## `autofix_class` and `owner` (routing signal only — this skill applies nothing)

| `autofix_class` | Meaning |
|-----------------|---------|
| `gated_auto` | A concrete `suggested_fix` is proposed; a caller may apply it after their own judgment. |
| `manual` | Actionable work needing design input or a cross-cutting decision before code changes. |
| `advisory` | Report-only — learnings, residual risk, rollout notes. |

| `owner` | Meaning |
|---------|---------|
| `downstream-resolver` | A caller or human should act after review. (Default for actionable findings.) |
| `human` | Judgment required before implementation. |
| `release` | Operational / rollout follow-up. |

Prefer `gated_auto` with a defensible `suggested_fix` for a localized change; `manual` when the fix
depends on product intent or architecture; `advisory` when nothing breaks. Set `requires_verification:
true` whenever the likely fix needs targeted tests or a follow-up validation pass.

**On `suggested_fix`:** propose one whenever any defensible code change is reachable from the diff,
cited code, a parallel pattern in the repo, or a framework convention. Be concrete (name the guard, the
helper, the value), not generic. Imperfect information is not grounds for omission — propose the most
defensible default and name the assumption. Omit only when there is genuinely no code-level change to
propose (the finding is a question, or the resolution is purely organizational).

## What you return

Return **one JSON object** as your final message to the lead — no prose around it, no file writes:

```json
{
  "reviewer": "<your persona name, e.g. 'security'>",
  "findings": [
    {
      "title": "Short, specific, <=10 words",
      "severity": "P0|P1|P2|P3",
      "file": "relative/path/from/repo/root",
      "line": 42,
      "why_it_matters": "Observable impact first — what breaks and who's hit — then the minimum code grounding. 2-4 sentences.",
      "autofix_class": "gated_auto|manual|advisory",
      "owner": "downstream-resolver|human|release",
      "requires_verification": true,
      "suggested_fix": "Concrete minimal fix, or omit if none is defensible",
      "confidence": 75,
      "evidence": ["path:line -- verbatim motivating line (first, required at 75/100)", "..."],
      "pre_existing": false
    }
  ],
  "residual_risks": ["risks you noticed but could not confirm as findings"],
  "testing_gaps": ["missing coverage you identified"]
}
```

`evidence` is an array with at least one item; at anchor 75/100 the first item is the quoted motivating
line. `why_it_matters` leads with observable behavior, not code structure, and stays tight. If you find
nothing, return an empty `findings` array (still populate `residual_risks` / `testing_gaps` if relevant).

**Intent check:** compare the code against the stated intent (and PR title/body when present). Code that
does something intent doesn't describe, or fails to do something intent promises, is a high-value finding.
